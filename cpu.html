Multicycle CPU (Parts 4B → 4C)

SystemVerilog · Multicycle datapath + controller FSM · Incremental ISA support

Datapath + Control Split FSM Controller R-type + I-type lw/sw beq/bne/j/jal/jr

Overview

Built a multicycle CPU by separating a reusable datapath from an FSM-based controller. Part 4B implements the baseline execute/writeback flow for R-type instructions, and Part 4C extends the same design to support immediates, memory access, and control flow (branches and jumps).

Part 4B — Baseline CPU (R-type core)

Architecture

  • Top-level cpu wires controller ↔ datapath and exposes debug signals (IR + regs)
  • Controller FSM cycles: FETCH → DECODE → EXEC_R → WB_R → FETCH
  • ALU operation selected by Funct decode in EXEC_R

See the Part B controller state sequencing and Funct→ALUControl mapping.

Supported R-type ops

add, sub, and, or, xor, nor, sll, srl, sra, slt

Controller sets ALUSrcA=1, ALUSrcB=00 and chooses ALUControl via Funct, then writes back rd. 

// Part B controller state flow (excerpt)
S0_FETCH:  next = S1_DECODE;
S1_DECODE: next = (Opcode == `OP_RTYPE) ? S6_EXEC_R : S0_FETCH;
S6_EXEC_R: next = S7_WB_R;
S7_WB_R:   next = S0_FETCH;

Part 4C — Extensions (ISA + memory + control flow)

New instruction support

  • I-type ALU: addi, andi, ori, xori, slti
  • Memory: lw / sw via address compute + memory stage + writeback
  • Control flow: beq, bne, j, jal, jr

Key datapath behaviors

  • Immediate generation: zero-extend for andi/ori/xori, sign-extend otherwise
  • Branch target precompute in DECODE (PC + imm<<2) stored in ALUOut
  • PC enable: PCWrite | (Branch & takeBranch), where takeBranch handles beq/bne
  • JAL writes $ra with PC+4 while also performing jump
// Part C branch control (excerpt)
assign takeBranch = ((Opcode == `OP_BEQ) & Zero) |
                    ((Opcode == `OP_BNE) & ~Zero);
assign PCEn = PCWrite | (Branch & takeBranch);
